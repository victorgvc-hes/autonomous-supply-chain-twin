[SLIDE 2: PROBLEM FORMULATION - 2 minutes]

THE TECHNICAL CHALLENGE:

We're forecasting intermittent demand - a notoriously difficult problem in 
time series. Here's why:

SPARSE DATA:
- 61.8% of observations are zeros
- Non-zero values follow long-tailed distribution
- Max:mean ratio of 65:1 (extreme variance)

VIOLATIONS OF STANDARD ASSUMPTIONS:
- Not stationary (trend breaks, level shifts)
- Not normally distributed (zero-inflated)
- Autocorrelation is weak and inconsistent
- Seasonal patterns exist but are noisy

BUSINESS CONSTRAINT:
- Can't just predict zeros (trivial but useless)
- Can't ignore zeros (they're the majority)
- Need probabilistic view, not just point forecasts

DATASET CHARACTERISTICS:
- M5 Walmart Competition dataset
- 10 products (FOODS category)
- 1,885 daily observations per product (2011-2016)
- 61 engineered features
- Train/test split: 1,857 train, 28 test days

The question: Can we beat naive baselines for this sparse data?

[SLIDE 5: EVALUATION FRAMEWORK - 1.5 minutes]

METRICS USED:

PRIMARY: RMSE (Root Mean Squared Error)
- Why? Penalizes large errors (important for stockouts)
- Formula: sqrt(mean((y_true - y_pred)^2))
- Baseline to beat: 0.7759 (MA-7)

SECONDARY: MAE (Mean Absolute Error)  
- More interpretable in business units
- Less sensitive to outliers than RMSE

TERTIARY: MAPE (Mean Absolute Percentage Error)
- Problem: undefined for zero actuals (common in our data)
- Used cautiously, not for ranking

CROSS-VALIDATION STRATEGY:
- Time series split (no random shuffling!)
- Train: days 1-1857
- Test: days 1858-1885 (last 28 days)
- No data leakage from future to past

WHY 28 DAYS?
- Standard retail planning cycle (4 weeks)
- M5 competition standard
- Balances statistical power vs. recency

MULTI-PRODUCT EVALUATION:
- Averaged RMSE across 10 products
- Best model per product identified
- No single winner → ensemble needed

[SLIDE 7: DIGITAL TWIN SIMULATION ARCHITECTURE - 2.5 minutes]

We built a custom Discrete Event Simulation (DES) engine.

WHY NOT USE EXISTING TOOLS?
- SimPy: Excellent but overkill for this use case
- AnyLogic: Commercial, not reproducible
- Custom: Full control, educational, production-ready

SIMULATION ARCHITECTURE:

CLASS: InventorySimulator
- State variables: inventory level, orders in transit
- Parameters: ROP, order quantity, lead time, costs
- Methods: simulate_day(), run_simulation(), get_kpis()

DAILY SIMULATION LOOP:
1. Receive orders (if arrival_day == today)
2. Fulfill demand (actual_demand from test set)
3. Check inventory level → trigger order if <= ROP
4. Calculate costs (holding + stockout)
5. Update state, log metrics

INVENTORY POLICY: Reorder Point (ROP)
- When inventory <= ROP → order fixed quantity Q
- ROP = avg_demand × lead_time + safety_stock
- Safety stock = z × σ × sqrt(lead_time)
- z = 1.65 for 95% service level target

PARAMETERS:
- Initial inventory: 10 units
- ROP: 1.0 units (forecast-based)
- Order quantity: 1.4 units (7-day supply)
- Lead time: 1 day
- Holding cost: $0.50/unit/day
- Stockout cost: $5.00/unit unmet

COST FUNCTION:
Total_Cost = Σ(holding_cost_t + stockout_cost_t)
           = Σ(inventory_t × h + stockout_t × s)

KPI TRACKING:
- Service level: fulfilled / total_demand
- Stockout rate: days_with_stockout / total_days
- Avg inventory: mean(inventory_t)
- Total cost: holding + stockout costs

[SLIDE 13: TECHNICAL Q&A PREPARATION]

Expected Technical Questions:

Q: "Why ARIMA over LSTM/Transformer?"
A: "Intermittent demand has insufficient signal for deep learning. LSTM needs 
    dense sequences. We tested TFT, encountered data sparsity issues. For 
    sparse data, statistical methods (ARIMA) are more parameter-efficient."

Q: "How do you handle concept drift?"
A: "Currently: retrain weekly on rolling window. Future: online learning with 
    exponential forgetting. Monitor prediction error over time, trigger retrain 
    when drift detected (e.g., error > 2σ threshold)."

Q: "What about seasonality decomposition?"
A: "Prophet handles this automatically with Fourier terms. For ARIMA, we tested 
    SARIMA but found weekly period (s=7) didn't improve RMSE significantly. 
    Yearly seasonality is weak in this data (r=-0.03 for month)."

Q: "How does simulation handle stochasticity?"
A: "Currently deterministic (uses actual test demand). Future: Monte Carlo 
    simulation with demand sampled from forecast distribution. This gives 
    confidence intervals on KPIs (e.g., '88.9% ± 5% service level')."

Q: "Computational complexity?"
A: "Feature engineering: O(n) per product. Model training: O(n log n) for ARIMA, 
    O(n) for MA. Simulation: O(T) where T=days. Total: O(n log n) per product. 
    Parallelizable across products. Scales linearly with cores."

Q: "How do you prevent data leakage?"
A: "Strict temporal split. No shuffle. Features use only lagged values. 
    Rolling calculations use .shift() to prevent lookahead. Test set never 
    seen during training. Validated with assertion checks."

Q: "Why not use XGBoost/LightGBM?"
A: "Tree-based models require dense features and large sample sizes. With 
    68% zeros and only 1,885 observations, we risk overfitting. Also, trees 
    struggle with extrapolation (forecasting beyond training range). ARIMA 
    is more suited to sparse time series."
